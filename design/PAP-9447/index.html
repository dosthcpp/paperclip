PAP-9447 · Wireframes

Permissions & controls — UX review

Lo-fi paired desktop + mobile wireframes for the five permission surfaces in the approved PAP-3631 plan: company access, scoped task assignment, agent permission detail, denied/read-only states, and the privacy rule.

Visual style is deliberately monochrome — the goal is to argue structure, copy, and the permission model, not pick a colour palette. The product should land inside the existing Settings & Agents surfaces with no new framed panels.

Design note

What we are arguing

Core flows

  • Company access lives where it already does (/company/settings/access) — but the page now has a peer Agents table next to Humans, and the grants column surfaces the new scope summary chip in line.
  • Scoped task assignment grants are a dedicated editor surfaced from the existing member-edit dialog. Three constraint families — project, managed-subtree, target-agent allowlist — sit as their own sections with radio + chip pattern, plus a live "all-of" summary in a right rail.
  • Agent permission detail consolidates the five SPEC 9.4 lenses into one tab on the agent page: profile visibility, config visibility/redaction, assignment/invocation, tool/secret policy, escalation. Each card is a single rule, not a settings tree.

Empty, error, and denied states

  • Out-of-scope picker rows stay visible but disabled with an "out of scope" trailing label — operators see the universe but can't act on what they don't own.
  • Denied toast and activity log entry use a single generic shape: actor · action · outcome. They never name the failed constraint, the target id, or the hidden project.
  • Read-only config renders fields at 0.7 opacity with a single banner ("View only · Board can edit this") and a "Request edit from board" affordance. Secret refs are visibly redacted with <redacted> placeholder text.
  • Empty state for a member with no scope: the grants cell reads "No explicit grants — implicit from role"; the editor's right-rail summary reads "Any project · Any subtree · Any agent".

Privacy rule (load-bearing)

  • The privacy page leads with one block: "Agent privacy does not hide work." Hidden vs. visible is a two-column list — agent-identity surface (hidden) on the left, work objects (visible) on the right.
  • The footer block names what is NOT covered (project/issue ACLs) so operators don't mistake the toggle for ACL-style scoping. Roadmap link points at the Pro/Enterprise tier.
Step 1 · Company & access

1.Company access — humans & agents in one place

The existing /company/settings/access page grows a sibling Agents table and surfaces the new scope grant inline. Roles cheatsheet sits up top so operators don't need to open a doc.

Annotations

  • 1 — Visibility note. Product copy: "Roles and grants control who can act — not what they see." Help icon links to the full privacy page.
  • 2 — Humans table. Same shape as today, but grants column now shows scope summary chip when tasks:assign_scope is set.
  • 3 — Agents table is new. Compact columns: role, profile visibility, config visibility, who can assign-to it, who can invoke it.
  • 4 — Private agent badge inline with the agent's name in the directory. Dense dropdown pickers (Screen 4) use a lock icon instead — see revision changelog.
Why this changes: today the page only shows humans. The plan adds agent-side permissions, and the access page is where operators expect to compare. Mobile reflows to cards so role/status/grants stack vertically; the scope summary becomes a full-width pill row.
Step 2 · Scoped assignment

2.Scoped task assignment grant editor

Editor surfaced from the existing member-edit dialog when tasks:assign_scope is enabled. Three constraint families with radio toggles and chip multi-pickers; a right rail shows the live "all-of" result.

02-scope-editor.svg1280×900 · desktop
mobile375×1100

Annotations

  • 1 — "All chosen scopes must match" explainer. Pre-empts the most common misread (OR vs AND).
  • 2 — Project scope. "Any project" is the explicit unscoped state so the radio always has a non-empty selection.
  • 3 — Managed-subtree scope. Pickers select a manager root — recursive descent is spelled out in copy, not left to inference.
  • 4 — Target-agent allowlist. Chip multipicker; suggested order matches reporting chain.
  • 5 — Live result rail. Mirrors the back-end scope JSON so the operator sees the assertion the server will check.
Why this shape: the SPEC allows multiple constraint families ANDed. A flat form would feel like one rule; this layout teaches the AND by structure. Mobile collapses the rail into a card directly above the sticky save button so the summary remains the last thing read before commit.
Step 3 · Agent detail

3.Agent permissions — the five lenses

A new Permissions tab on the agent detail page. Each SPEC 9.4 lens lives as one card with one rule. The Private toggle sits at the top with the privacy rule restated inline so the consequence is unambiguous.

03-agent-permissions.svg1280×1320 · desktop
mobile375×1700

Annotations

  • 0 — Private marker + privacy rule restated. Lead line is the product copy "Agent privacy does not hide work." with Learn more →. Same wording everywhere to avoid drift.
  • 1 — Profile visibility. Company-visible vs. Board-only — two states.
  • 2 — Config visibility & redaction. Secret-ref fields are always shown as <redacted> regardless of viewer.
  • 3 — Assignment / invocation. Two distinct principal lists with chip pickers.
  • 4 — Tool / secret policy. Allowed tools and secret refs as labeled chips, with a redaction notice attached.
  • 5 — Escalation chain — manager → grand-manager → board. No SLA copy on hops (escalation is event-driven, not time-boxed).
Why one tab: these five lenses are easy to confuse. Surfacing them as siblings with consistent card chrome and a recent-changes audit at the bottom makes drift visible.
What we're avoiding: a settings tree with nested toggles. Every nest is a place a permission gets misread. Cards stay flat.
Step 4 · Failure states

4.Denied and read-only — the no with no leakage

Four patterns that all share one rule: name the unblock owner, never the missing resource. SPEC 9.8 returns 403 with a generic scope message; the UI wraps that into a single calm sentence and a "Request wider scope" affordance.

04-denied-readonly.svg1280×1320 · desktop
mobile375×1620

Annotations

  • 1 — Picker shows in-scope agents at normal weight and out-of-scope agents at 45% opacity with an "out of scope" trailing label. Disabled, not hidden — the operator sees the universe but can't act.
  • 2 — Copy guidelines: one sentence + a named owner. Never list hidden agents or projects.
  • 3 — Prohibited copy: things that leak resource shape. These are what NOT to ship.
  • 4 — Read-only config: fields render at 0.7 opacity; banner is the only colour-free signal that this is a permissions state, not a loading state.
  • 5 — Secret redaction explainer. Same rule applies to UI, activity log, and exports.
  • 6 · 7 — Denied toast + activity entry use the same shape. Actor · action · outcome (scope). No target id.
Anti-patterns we're rejecting: "Project X is not in your allowed projects" — leaks that Project X exists and that you have an allowed-list. The wireframes call these out inline.
Step 5 · The privacy rule

5.Privacy rule — what "private" actually means

A dedicated Privacy page on the agent detail, designed so the toggle and its meaning sit on the same screen. The two-column "what gets hidden / what stays visible" list teaches the boundary by example.

05-privacy-rule.svg1280×1080 · desktop
mobile375×1500

Annotations

  • 1 — Toggle row. Sub-copy spells out the two effects (directory hiding, config readability).
  • 2 — The rule, restated as a large block — same wording as the agent-permissions tab banner.
  • 3 — Hidden column. Bullets are agent-identity surface only.
  • 4 — Visible column. Issues, comments, projects, costs, activity. The "why this distinction" callout names the board accountability surface as the reason.
  • 5 — "What this doesn't do" strip with dashed border. Names project/issue ACLs and points at the Pro tier so operators don't conflate.
Why a dedicated page: the rule is load-bearing for the plan and easy to misread from a single toggle label. The page exists to anchor that the marker is identity-only, not work-object visibility. It also keeps the agent-permissions tab uncluttered.
Revision 2 · 2026-05-18

What changed in this revision

Board accepted Q1, Q2, Q4 as proposed and asked for targeted revisions on the rest. This revision is the focused pass before final approval.

Targeted fixes

  • Lock icon in dropdowns (Screen 4). Dense picker rows now show a compact lock icon (with accessible title/aria-label) instead of a text "private" badge. The full text badge stays in directory views (Screens 1, 3) where row density allows it. Added a private agent (Vega) into the in-scope group so the pattern is visible.
  • "V1" copy removed from user-facing UI. All in-product strings that referenced V1 are now plain product language. Section labels like "NOT IN V1" → "WHAT THIS DOESN'T DO"; section headlines like "The V1 rule" → "Agent privacy does not hide work."; body copy "In V1, scope at the role/grant layer" → "Scope at the role/grant layer". The review-site annotation copy was scrubbed at the same time for consistency.
  • Meta copy replaced with product copy + help affordances (Screens 1, 3, 5). Headline "Visibility in V1 is company-scoped" → "Roles and grants control who can act — not what they see." Added a help (?) icon on the Company Access banner and Agent permissions private-marker card. Each surface that mentions privacy carries a Learn more → link that routes to the Privacy page.
  • SLA copy removed from escalation (Screen 3). Manager hop now reads simply "manager"; mobile shows "CTO · Atlas · manager". Escalation is event-driven, not time-boxed in this slice.
  • Denied toast names the owner action (Screen 4). Both desktop and mobile toasts now include an explicit "Ask Maya Chen" button next to Dismiss, mirroring the picker-side "Ask Maya Chen" affordance. The button opens a draft comment on the failing task. Toast body copy ("Ask Maya Chen if this should change.") preserves the named owner in text for screen readers and ambient context.

Preserved from revision 1 (board approved)

  • Q1 — Agents as a peer table on Company Settings → Access.
  • Q2 — Mobile scope summary as a card above sticky save.
  • Q4 — Out-of-scope picker rows visible as disabled / labeled, not hidden.

Open questions for this round

  • Help destination. The "Learn more →" links currently point at the Privacy page (Screen 5). If we want a single product-docs URL instead, flag and we'll route there. The wireframes don't depend on the destination.
  • Lock icon stroke weight. The dropdown lock uses 1.4pt stroke to read at 13–14px row height. If accessibility audit later asks for a higher-contrast monochrome glyph, the icon can be swapped for a filled variant without re-laying out the row.
Wireframes follow the in-tree wireframe skill house style — black on white, 1.5pt strokes, 8px grid, type sizes 12/14/20/28. Red dashed numbered callouts are the annotation layer only and not part of the proposed UI. Source: PAP-9447 · feeds PAP-9446 review.
